# Video Downloader Pro App 全面回归测试方案

更新日期：2026-05-06

## 结论先行

当前产品方向应该采用“持久会话 + 可解释恢复”的下载器模型：App 重新打开后保留上次未完成任务是合理的，但 UI 必须清楚告诉用户哪些是上次遗留、哪些已完成、哪些可续传、哪些需要重新下载。重复导入同一张表格时，不应该静默重复创建任务，也不应该只提示“已添加”。正确反馈是“新增 N 个、识别已有 M 个、已完成 X 个、可续传 Y 个、失败 Z 个”。

参考行业实践：

- aria2 使用 `--input-file`/`--save-session` 保存队列，`--continue` 续传已有部分文件，`--max-concurrent-downloads` 控制队列并发。
- GNU Wget 的 `--continue` 模式基于本地已有部分文件继续拉取，但依赖服务端 Range 支持。
- Windows BITS 明确把下载任务作为持久 job 管理，并区分 transient/fatal 错误、重试间隔和最终完成/取消。

## 状态设计基线

| 场景 | 推荐行为 | 当前验证点 |
| --- | --- | --- |
| App 启动发现上次 `downloading` | 不自动继续下载，转换为 `paused` 或“可恢复”，等待用户明确操作 | 后端 `load_persisted_state` 已转换为 `Paused` |
| App 启动发现 `.part`/resume 快照 | 显示为可续传，保留已下载字节和进度 | 后端 `hydrate_existing_file_state` 会读取文件大小和 resume 快照 |
| App 启动发现最终文件 + 完成标记 | 标为 `completed`，进度 100% | 后端 `.vdstate` marker 已支持 |
| 重复导入同一表格 | 不重复创建；合并已有任务并展示分类摘要 | 需前端展示准确 reconciliation |
| 已完成文件无完成标记 | 不能仅凭同名文件武断完成，除非有文件大小/哈希/业务 ID 证据 | 保持谨慎，避免误判 |
| 并发数从高调低 | 不杀死已在跑任务，新并发目标对后续调度生效 | 后端 semaphore 使用 deferred reduction |
| 并发数从低调高 | 队列未暂停时立即补位启动等待任务 | 后端 `process_task_queue` 已调用 |

## 测试数据准备

1. 使用用户给的 `/Volumes/soft/10-codex/006-tauri-video-batch-downloader/sqlResult_1.xls`。
2. 准备本地下载目录：
   - 空目录：`/tmp/vdp-e2e-empty`
   - 已完成目录：包含一个目标视频文件和对应 `.vdstate`
   - 半下载目录：包含同名目标文件或 `.part`/resume 快照
   - 无权限目录：去掉写权限，用于验证错误提示
3. 准备 3-5 个小体积公开视频 URL 或本地 HTTP 测试服务，覆盖 200、404、429、断连、无 Range 支持。

## 每轮真实回归顺序

1. 运行静态检查：`pnpm type-check && pnpm lint && pnpm exec vitest run`。
2. 运行 Rust 检查：`cargo fmt --manifest-path src-tauri/Cargo.toml --all --check`、`cargo clippy --manifest-path src-tauri/Cargo.toml -- -D warnings`、`cargo test --manifest-path src-tauri/Cargo.toml`。
3. 启动真实桌面 App：`pnpm tauri dev`。
4. 执行 P0-01 到 P0-12，截图或记录每个失败点。
5. 按失败严重度修复：数据损坏/覆盖 > 状态错误 > 错误提示 > 视觉细节。
6. 修复后跑 Graphify smart sync：`./scripts/graphify-sync.sh smart`。
7. 使用 GitNexus 做影响检查，确认改动没有扩大到无关模块。

## 当前已确认的待优化点

1. 重复导入后前端提示会把后端返回的已有任务算成“新增”，已改成 reconciliation 摘要。
2. App 启动恢复上次未完成任务的策略是合理的；已修复启动后自动排队下载问题，重开后保留任务和断点，但队列保持暂停。
3. 已完成识别依赖 `.vdstate`，这是安全方案；后续可增加可选 hash/ETag 校验来提升跨会话识别准确率。
4. 并发调低不应中断已运行任务；现有后端策略正确，必须增加真实 App 用例和单元/集成覆盖。
